Adding Zend Search Lucene to a Silex project

Tags: php, search, silex
Development
Sunday, February 24th, 2013
Tweet

Recently I added a search feature to Archie’s Recipes using Zend Search Lucene. The site is semi-static, relying upon the PHP micro-framework Silex to format static HTML files with layout templates (using Twig), provide a simple email contact form, and add a few simple navigational features (discussed here previously).

Zend Search Lucene natively supports indexing of HTML documents. Since all of the content of the site is stored in HTML files, adding search was pretty easy.

  1. Add Zend Search Lucene to the Silex project
  2. Script the creation of a search index
  3. Create a Service Provider to handle search queries
  4. Create a route, controller, and a page template to handle search requests
  5. Update the search index with each site update

Just a little Zend

Zend Search Lucene hasn’t been ported over to Zend Framework version 2 yet, which means you can’t install it using composer. Fortunately though, Zend Framework is written in a modular fashion allowing developers to use portions of the library without storing the entire framework.

I downloaded the most current version of Zend Framework 1 (1.12 at the time of this writing) and moved it into the ‘src’ directory of the project. Since I only needed search, I deleted almost all of Zend Framework, leaving the following:

/src
    /Zend
        Exception.php
        /Search

Silex configures an auto-loader for PHP classes stored in the ‘src’ directory which follow standard PHP naming conventions. Zend does this, so the autoloader works just fine…mostly. Zend Framework 1 uses PHP ‘require’ statements throughout; the framework doesn’t assume an autoloader is present. These require statements will throw errors unless the PHP include path is configured correctly. To solve this I added the following line near the end of the composer.json file which tells Composer to manually set an include path for legacy projects:

"include-path": ["src/"]

Now ZF can find itself with both hands.

This feature is deprecated, it exists to allow composer to work with legacy projects like Zend and Smarty. It may go away eventually, so be warned.

I chose to drop ZF into the ‘src’ directory instead of the ‘vendor’ directory because:

  • I’ve customized the framework by deleting 95% of it – so now it’s effectively my own application code and not a library.
  • Zend Framework isn’t managed by composer while the rest of Silex is. If I run a composer update on my project, and ZF is stored in the ‘vendor’ directory, composer may not be happy.

Create a search index

Zend Search Lucene stores its search index in the local file system. I created a ‘data’ directory to store it and a ’scripts’ directory to store the index creation scripts. These names are nice and generic so I can other utilities here if needed. A typical Silex file system layout with these additions might look like this:

/data
    /search_index
/scripts
    createIndex.php
/src
/vendor
/web

The search index is stored in the ‘search_index’ directory under ‘data’ (obviously). The ‘createIndex.php’ is a simple command-line PHP script which does the following:

  1. Defines the locations of the content and the search index
  2. Deletes the old search index, if it exists
  3. Creates a new search index
  4. Indexes each HTML file in the content directory

There’s nothing really special here, beyond normal PHP file handling and directory reading. The only thing special to point out is that I add a custom ‘url’ field to each HTML document as its being indexed. This way, when a file is identified by a search query, the URL for the corresponding page is available in the search result with no further work.

// create the index
$idx = Zend_Search_Lucene::create('/path/to/search/index');
// create document for indexing
$doc = Zend_Search_Lucene_Document_Html::loadHTMLFile('/path/to/doc.html');
// add non-searchable url field
$doc->addField(Field::unIndexed('url',$path));
// index doc
$idx->addDocument($doc);

An ‘unIndexed’ field type stores the field value in the search index, but doesn’t make it available for searching. The Zend_Search_Lucene_Document_Html class automatically converts HTML title, keywords, and other meta tags into their appropriate search fields.

Create a search provider

Silex’s dependency injection system allows for developers to create and register service providers in their application. This makes it easy to share custom code throughout a Silex application. In this case, I needed a small service to perform a search against the Lucene index with a provided query.

When the service is registered a path to the search index is provided as a configuration option. The service itself is a simple closure which opens the search index and performs a search for the provided query. The search results are formatted into a tidy array which can be dropped right into a page template without further processing. 

<?php

namespace Abouthalf;

use Silex\ServiceProviderInterface;
use Silex\Application;
use Zend_Search_Lucene as Search;
use Zend_Search_Exception as SearchException;
use Zend_Search_Lucene_Search_QueryParser as Parser;
use Zend_Search_Lucene_Document_Html as HtmlDoc;
use Exception;

/**
 * Provider for querying Zend_Search_Lucene index
 */
class SearchProvider implements ServiceProviderInterface
{
    const SEARCH_PROVIDER = 'search';
    const SEARCH_INDEX = 'search.index';

    /**
     * Register query service
     *
     * Load index from config parameters, query, return results
     *
     * @param Application $app An Application instance
     * @throws \Exception
     * @return \Zend_Search_Lucene_Search_QueryHit[]
     */
    public function register(Application $app)
    {

        $app[self::SEARCH_PROVIDER] = $app->protect(function($query) use ($app) {
            $indexPath = $app[SearchProvider::SEARCH_INDEX];
            if (!$indexPath)
            {
                throw new Exception(__CLASS__. ' requires a valid search.index path');
            }

            $index = SearchProvider::getIndex($indexPath);
            if ($index)
            {
                return SearchProvider::query($index, $query);
            }
            else
            {
                return array();
            }
        });
    }

    /**
     * Bootstraps the application. Required by interface
     */
    public function boot(Application $app) {}

    /**
     * @param $path
     * @return \Zend_Search_Lucene_Interface|null
     */
    public static function getIndex($path)
    {
        try
        {
            return Search::open($path);
        }
        catch(SearchException $e)
        {
            error_log($e->getMessage());
            return null;
        }
    }

    /**
     * Find, format, return results
     *
     * @param \Zend_Search_Lucene_Interface $index
     * @param string $query
     * @return array
     */
    public static function query(\Zend_Search_Lucene_Interface $index, $query)
    {
        $q = Parser::parse($query);
        /* @var $hits \Zend_Search_Lucene_Search_QueryHit[] */
        $hits = $index->find($q);
        $results = array();
        foreach($hits as $hit)
        {
            $h = array();
            $h['title'] = $hit->getDocument()->getFieldValue('title');
            $h['url'] = $hit->getDocument()->getFieldValue('url');
            $results[] = $h;
        }
        return $results;
    }
}

Registering the provider with the application is simple:

/**
 * Search service
 * */
$app->register(new SearchProvider(), array(
        SearchProvider::SEARCH_INDEX => __DIR__.'/../data/search_index'
    ));

Now the Silex application can perform a search from any controller and route.

Search route and controller

All of the work of searching the index is handled by Zend Search Lucene. The search provider hides the work of loading and searching the index and provides a conveniently formatted array of search results. The controller function need only pass the search query along to the search service and place the results into the page template.

/**
 * Search controller
 *
 */
$app->get('/search', function(Application $app, Request $request)
{
    $query = $request->get('q',null);
    $out = array();
    $out['query'] = str_replace('"','',$query);
    if ($query)
    {
        $out['hits'] = $app[SearchProvider::SEARCH_PROVIDER]($query);
    }

    /** @var $twig Twig_Environment */
    $twig = $app['twig'];
    return $twig->render('search.twig',$out);
});

The search results are then displayed in a simple list:

{% if hits %}
<h1>Search results</h1>
<h2>Pages and blog posts matching:<br />“{{ query }}”</h2>
<ul>
    {% for hit in hits %}
    <li>
        <a href="{{ hit.url }}">
            {{ hit.title}}
        </a>
    </li>
    {% endfor %}
</ul>
{% endif %}

Updating the index with a Git post-hook

I’m using Git for version control and to deploy updates to Archie’s Recipes. Each time a change is committed to the repository, a "post-receive” hook exports the project to the right place on the server. To update the search index I have the hook run the ‘createIndex.php’ script after the site is exported. Git “hooks” are just shell scripts executed by Git when particular actions occur. In this case I added something like ‘php /path/to/scripts/createIndex.php;’ to the hook script after the check out command. This way each and every content update gets a fresh, new search index so the site search tool is never out of date.

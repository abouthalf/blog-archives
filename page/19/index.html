No. No tables weren’t easier.

Tags: css, html
Development
Sunday, December 16th, 2012
Tweet

I ran across this funny tweet in my timeline the other day. 

<table>
        <tr>
            <td>
                <h1>1999

<div class=“container”>
    <div class=“row”>
        <div class=“span6”>
            <h1>2010

via @bobearth

I’m not going to lie; I LOLed. But, it stuck in my mind like a pebble stuck in my sock. Because, while anyone who actually made websites in the 90’s can relate, it’s not really a fair comparison.

The joke is basically this:

Look how far we haven’t come. Instead of using tables for layout, we now use an equivalent number of DIV elements and CSS to simulate table-like layout.

So it’s a joke, not to be taken seriously.

But seriously though? Back in the table days, that table would look like this:

<table cellpadding=“0” cellspacing=“0” border=“0”>
    <tr>
        <td valign=“top” align=“left”>
            <h1>1999</h1>
        </td>
    </tr>
</table>

See, back in the bad old days, browsers didn’t agree on the default display of a table. If you wanted to prevent display problems across browsers and platforms, you had to get specific. And you couldn’t do it in a global CSS file, you had to do it in each and every HTML document.

So you have your “simple” table layout. What if we want to add an additional row to this example layout, perhaps one where the grid columns / cells have a slightly different configuration? Or what if you needed an empty column for spacing purposes? 

<table cellpadding=“0” cellspacing=“0” border=“0”>
    <tr>
        <td>&nbsp;</td>
        <td valign=“top” align=“left” colspan=“2”>
            <h1>Party like it’s 1999</h1>
        </td>
        <td valign=“top” align=“left”></td>
        <td>&nbsp;</td>
    </tr>
    <tr>
        <td valign=“top” align=“left”></td>
        <td valign=“top” align=“left”></td>
        <td valign=“top” align=“left”></td>
        <td valign=“top” align=“left”></td>
        <td valign=“top” align=“left”></td>
    </tr>
</table>

“colspan” for the win. Where by “win” I mean “fail”. Oh, and you had to have that non-breaking space there, because early browsers hid empty table cells and no the CSS rule empty-cells: show didn’t work. Without that non-breaking space (or a comment, or something) the cell wouldn’t display, and a big chunk of your design might just disappear. These were the days of Internet Explorer 5 after all.

The div-float-grid approach to this would be simpler, though not great.

<div class=“container”>
    <div class=“row”>
        <div class=“span5 push1”>
            <h1>2010</h1>
        </div>
        <div class=“span3 pull1”>
        </div>
    </div>
    <div class=“row”>
        <div class=“span2”></div>
        <div class=“span2”></div>
        <div class=“span2”></div>
        <div class=“span2”></div>
        <div class=“span2”></div>
    </div>
</div>

It is less verbose, but there is still a lot of junk. It is an improvement, however. A significant one. Because <divs> aren’t table cells, you can have alternate stylesheets which reformat the horizontal grid into a vertical column for easy reading on handheld devices and printing[1]. Google can tell the difference between <DIV> soup and actual tables full of data. Screen readers can work around this fairly well. It’s not great, but it’s definitely better.

Sure, it could be improved with a little housecleaning. One could map all the presentation-specific CSS classes to something more semantically meaningful. If you’re smart about how you define your grid system in your CSS, you can probably get rid of the <div class=“row”> elements. But at least this mess can — hypothetically — be accessible and device independent. Old school tables can not.

Big grid systems, as found in Twitter’s Bootstrap framework, exist to help solve the problem of marking up big complex layouts for applications or commercial sites. Bootstrap is a messy solution to a messy problem. But most sites are simpler. Your classic two-column blog layout, today, looks like this:

<header>
    <h1>My Blog</h1>
</header>
<nav>
    <!-- navigation --> 
</nav>
<article>
    <!-- article -->
</article>

Whereas in the bad old days, this mess was common:

<table cellpadding=“0” cellspacing=“0” border=“0” width=“100%”>
    <tr>
        <td valign=“top” align=“left” colspan=“2”>
            <h1>My Web Ring</h1>
        </td>
    </tr>
    <tr>
        <td align=“left” valign=“top” width=“100”>
            <!-- navigation --> 
        </td>
        <td align=“left” valign=“top”>
            <!-- article -->
        </td>
    </tr>
</table>

The former is trim, flexible, and semantically meaningful while the latter is locked into one display mode, heavy, clunky, and broken.

So, yes, big heavy CSS grid systems repeat some of the mistakes of table-based layouts – but those systems are an option and aren’t the only way to do things. Plus, they’re infinitely more flexible than their table-based inspiration.

Don’t forget how good we HTML5 developers have it. You damn kids.

  1. Modern browsers might let you reshape tables for other display types with CSS…but that’s just crazy-pants behavior.  ↩
